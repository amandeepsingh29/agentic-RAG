RAG Documentation Assistant

Project wiki · knowledge base, architecture, evaluation, and scope

RAG Documentation Assistant answers questions from a local collection of official Kubernetes documentation. It is a bounded Agentic RAG system: it retrieves evidence, evaluates it, rewrites weak queries when useful, and refuses when the evidence is not sufficient.

Result: each approach wins a different problem
In a 14-question live LLM comparison, Agentic RAG scored highest overall at 78.6% because it recovered a weak storage query that classic RAG refused. Classic RAG tied agentic RAG on ordinary in-corpus questions while running faster. No-RAG LLM scored best on general questions outside this corpus, but it produced no project citations and failed unsupported airline-policy safety.

What this solves: it reduces unsupported answers, makes retrieved evidence visible to users, and gives developers a trace of how the system reached an answer or refusal.

What it solves

Unsupported questions

Prevents the model from filling gaps with general knowledge by refusing when the corpus does not support an answer.

Weak retrieval

Uses a bounded query rewrite when a question is ambiguous or the first evidence set is weak.

Trust and debugging

Shows source chunks and a step trace so a user or engineer can inspect the answer path.

How to read this: no-RAG is useful when broad pretrained knowledge is acceptable, but it cannot guarantee scope or cite the company corpus. Classic RAG is best when retrieval is straightforward and low latency matters. Agentic RAG is best when questions are ambiguous or the first retrieval is weak. The result is a tradeoff, not proof that one architecture is universally better.

Architecture

  1. Documents: official Kubernetes Markdown pages are stored locally.
  2. Chunking: documents are cleaned and split into overlapping passages.
  3. Embeddings: the same all-MiniLM-L6-v2 model embeds passages and questions.
  4. Vector store: PostgreSQL stores vectors, content, and metadata; pgvector performs cosine retrieval.
  5. Agent control: the agent retrieves, evaluates evidence, and may rewrite the query up to three iterations.
  6. Generation: only selected context is sent to OpenRouter tencent/hy3:free.

Grounding and refusal

The pipeline refuses before generation when there are no useful hits, similarity is too low, or retrieved text has insufficient overlap with the question. The generation prompt also instructs the model to answer only from supplied passages.

Answers include source filenames, titles, chunk IDs, positions, and retrieval scores. The API does not expose absolute server paths.

Agent behavior

The controller in agentic_rag/agentic_pipeline.py is intentionally bounded. It records retrieve, evaluate_evidence, rewrite_query, and refuse actions, then stops after at most three retrieval iterations.

In the benchmark, the question “How does this work for storage?” was refused by classic RAG but recovered by agentic query rewriting. The agent does not browse the internet, call arbitrary tools, run an unbounded planner, or use self-reflection.

Limitations

  • The evaluation has 14 labeled questions and is a snapshot, not a production quality guarantee.
  • Keyword checks and expected refusal labels are transparent proxies, not a human or independent LLM quality judge.
  • The selected corpus is not a complete Kubernetes or enterprise knowledge base.
  • Similarity and term overlap are safety heuristics, not factual verification.
  • The free hosted model can be slow, unavailable, or change behavior.
  • Private company documents require access-control metadata and tenant-aware filters.

Usage

Open the documentation chat for user testing. Run rag-docs compare --documents data/kubernetes --questions data/eval_queries.jsonl to regenerate the comparison report. The browser chat runs on port 8080.
