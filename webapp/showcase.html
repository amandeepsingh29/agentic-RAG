RAG Documentation Assistant

Project knowledge base and implementation reference

RAG Documentation Assistant is an agentic retrieval-augmented generation system for answering questions from a local corpus of official Kubernetes documentation.

Architecture. Documents → chunking → embeddings → PostgreSQL with pgvector → agent-controlled retrieval and evidence evaluation → OpenRouter generation. The agent is bounded to three retrieval iterations and cannot answer without sufficient evidence.

The browser chat is the user surface. Index refresh and ingestion remain backend and CLI operations so end users cannot choose arbitrary filesystem paths or rebuild the corpus accidentally.

Knowledge base

The current corpus contains 12 pages downloaded from the Kubernetes website repository and stored under data/kubernetes. The files are cleaned from Markdown presentation syntax before chunking.

Pipeline

  1. Ingestion: load Markdown, text, or PDF files and preserve source metadata.
  2. Chunking: split documents into overlapping passages with document and position identifiers.
  3. Embedding: encode chunks and questions with sentence-transformers/all-MiniLM-L6-v2.
  4. Storage: write vectors, content, and metadata to PostgreSQL with pgvector.
  5. Agent retrieval: retrieve evidence, evaluate the score and term overlap, and rewrite ambiguous or weak queries.
  6. Bounded control: stop after three retrieval iterations, answer when evidence is sufficient, or refuse.
  7. Generation: send only the selected retrieved context to OpenRouter model tencent/hy3:free.

pgvector design

The project uses a local PostgreSQL database named rag_assistant with the pgvector extension. Docker is not required for local development.

The knowledge_base table stores the embedding, chunk content, source metadata, and document metadata together. An HNSW index uses cosine distance for approximate nearest-neighbor search, while a GIN index supports metadata filtering.

The API accepts filters such as {"extension": "md"} on POST /ask. Keeping permissions and vectors in PostgreSQL gives the future internal assistant one place for SQL access checks, audit data, and retrieval.

Grounding policy

The pipeline refuses before generation when there are no hits, the top similarity score is below the configured threshold, or the question has insufficient term overlap with the retrieved context. The generation prompt tells the model to answer only from the supplied passages.

Successful responses include source filenames, document titles, chunk identifiers, positions, and retrieval scores. Absolute server filesystem paths are not exposed to users. A refusal returns I don't know. and no citations.

Agent behavior

The agentic controller is implemented in agentic_rag/agentic_pipeline.py. Each run records its actions as a trace containing retrieval, evidence evaluation, query rewriting, and refusal steps.

Query rewriting is deliberately bounded and evidence-driven. The controller uses retrieved document titles to clarify pronouns or weak questions, then searches PostgreSQL again. It does not call arbitrary tools, browse the internet, or run unbounded self-reflection.

Limitations

  • Local PostgreSQL requires the pgvector extension to be installed before startup.
  • The corpus is a selected 12-page Kubernetes set, not a complete enterprise knowledge base.
  • Similarity and term overlap are safety heuristics, not a guarantee of factual correctness.
  • The agentic loop increases latency and cost compared with the classic baseline.
  • The free hosted generation model may be slow, unavailable, or change behavior.
  • Private company documents need access-control metadata and tenant-aware filters before production use.

Usage

Open the documentation chat for user testing. Set OPENROUTER_API_KEY and DATABASE_URL, run ingestion with rag-docs ingest --documents data/kubernetes --index data/pgvector, and start the API on port 8080.
