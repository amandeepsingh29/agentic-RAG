RAG Documentation Assistant

Project knowledge base and implementation reference

RAG Documentation Assistant is a classic retrieval-augmented generation system for answering questions from a local corpus of official Kubernetes documentation.

Architecture. Documents → chunking → embeddings → Qdrant vector store → filtered retrieval → OpenRouter generation. The system is not agentic: it has no query planning, iterative retrieval, tool orchestration, or self-reflection loop.

The browser chat is the user surface. Index refresh and ingestion remain backend and CLI operations so end users cannot choose arbitrary filesystem paths or rebuild the corpus accidentally.

Knowledge base

Pipeline

  1. Ingestion: load Markdown, text, or PDF files and preserve source metadata.
  2. Chunking: split documents into overlapping passages with document and position identifiers.
  3. Embedding: encode chunks and questions with sentence-transformers/all-MiniLM-L6-v2.
  4. Storage: write vectors and payload metadata to the persistent Qdrant collection.
  5. Retrieval: run cosine similarity search, optionally restricted by payload filters.
  6. Generation: send only retrieved context to OpenRouter model tencent/hy3:free.

Qdrant design

The project uses Qdrant's embedded persistent client through QdrantClient(path="data/qdrant"). Docker and a separate Qdrant server are not required for the current development setup.

Each Qdrant point contains an embedding and a payload containing chunk_id, doc_id, source, title, text, position, and document metadata. The collection is named knowledge_base and uses cosine distance.

The API accepts filters such as {"extension": "md"} on POST /ask. The same design can later move to a Qdrant server or Qdrant Cloud without changing the application-level retrieval contract.

Grounding policy

The pipeline refuses before generation when there are no hits, the top similarity score is below the configured threshold, or the question has insufficient term overlap with the retrieved context. The generation prompt tells the model to answer only from the supplied passages.

Successful responses include source filenames, document titles, chunk identifiers, positions, and retrieval scores. Absolute server filesystem paths are not exposed to users. A refusal returns I don't know. and no citations.

Limitations

  • Qdrant local mode is persistent but intended for the current single-process development setup; deployment can later use Qdrant server or Cloud.
  • The corpus is a selected four-page Kubernetes set, not a complete enterprise knowledge base.
  • Similarity and term overlap are safety heuristics, not a guarantee of factual correctness.
  • The free hosted generation model may be slow, unavailable, or change behavior.
  • Private company documents need access-control metadata and tenant-aware filters before production use.

Usage

Open the documentation chat for user testing. Set OPENROUTER_API_KEY, run ingestion with rag-docs ingest --documents data/kubernetes --index data/qdrant, and start the API on port 8080.
